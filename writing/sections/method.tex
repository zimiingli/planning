\section{Direction-Informed Adaptive Learning}
\label{sec:method}

The analysis in Section~\ref{sec:signal-landscape} implies three
constraints for any cross-setting adaptive gate: it must discover
the signal--utility direction per (environment, backbone) pair, it
must draw from a multi-signal feature pool that approximates the
latent state type $\tau(s)$, and it must collect training data
without conditioning on the very signal whose direction is
unknown. DIAL satisfies these constraints through three stages
(Fig.~\ref{fig:method_overview}): randomized exploration
(\S\ref{sec:explore}), feature construction
(\S\ref{sec:reason}), and sparse gate fitting
(\S\ref{sec:learn}).

\subsection{Explore: Signal-Agnostic Data Collection}
\label{sec:explore}

Exploration collects the $(s, U)$ pairs used to fit the gate.
The central constraint is signal-agnosticity: the collection
policy must not condition on any signal whose semantics we are
trying to learn. A gate trained on uncertainty-triggered states
systematically under-samples exactly the states that determine
$\rho$'s sign, biasing the fitted direction toward the prior
assumption rather than testing it. DIAL therefore uses the
simplest policy consistent with signal-agnosticity. At each
step~$t$ in episode~$i$, a uniform random coin flip
$z_t^{(i)} \sim \mathrm{Bernoulli}(\varepsilon_{\text{explore}})$
determines whether to trigger the optimizer~$T$ ($z_t{=}1$) or
use the base action ($z_t{=}0$). Aggregating across
$N_{\text{explore}}$ episodes yields an approximately unbiased
dataset (the Bernoulli trigger eliminates selection bias; the
remaining approximation is the truncated-rollout estimator
of~$U$):
\begin{equation}
  \mathcal{D} = \bigl\{
    \bigl(\phi(s_t^{(i)}),\; U_t^{(i)}\bigr)
    : z_t^{(i)} = 1
  \bigr\}_{t, i},
  \label{eq:dataset}
\end{equation}
where $U_t^{(i)} = \mathbf{1}[V(a_T) > V(a_B)]$ indicates
whether the optimizer's selected action outperformed the base
action, as estimated by paired counterfactual rollouts from the
same state snapshot, and $\phi(s)$ is the feature vector described
in \S\ref{sec:reason}.

\paragraph{Estimating $U$.}
For each explored state with $z_t{=}1$, we estimate utility by
counterfactual evaluation from the same state snapshot: we fork
the environment, execute both the optimizer-selected action $a_T$
and the base action $a_B$ via truncated rollouts of horizon $H$,
and set $U_t = \mathbf{1}[V(a_T) > V(a_B)]$. This paired design
eliminates selection bias; the remaining approximation is the
truncation horizon. Per-environment estimation
details are in Appendix~\ref{app:environment}.

\subsection{Reason: LLM Feature Layer}
\label{sec:reason}

The next step builds the feature pool $\phi_{\text{cand}}(s)$
that the gate will be fit over. This pool must be rich enough to
include each environment's relevant signals, but not so
environment-specific that deployment requires manual engineering
per task.

DIAL's default is a fixed set of universal features:
\texttt{step\_count}, \texttt{token\_entropy},
\texttt{state\_length}, \texttt{action\_entropy}, and others.
These are available in every environment without task-specific
engineering. In the environments we study, this pool alone is
sufficient to recover a gate that outperforms all fixed-direction
baselines (\S\ref{sec:ablation}).

For environments where universal features may be inadequate,
DIAL offers an optional automation layer: an LLM is given a
summary of $\mathcal{D}$ and proposes task-specific feature
hypotheses $\phi_{\text{LLM}}(s)$, which join the universal pool
to form
$\phi_{\text{cand}}(s) = \phi_{\text{univ}}(s) \cup
\phi_{\text{LLM}}(s)$. The $\ell_1$ penalty in the gate fitting
stage (\S\ref{sec:learn}) automatically discards uninformative
proposals. Details of the LLM input format and prompt template
are in Appendix~\ref{app:llm-feature-layer}; the ablation in
\S\ref{sec:ablation} quantifies the contribution of this layer
per environment.

\subsection{Learn: Sparse Linear Gate}
\label{sec:learn}

Given $\mathcal{D}$ and $\phi_{\text{cand}}$, DIAL fits an
$\ell_1$-regularized logistic regression:
\begin{equation}
\mathbf{w}^*, b^* = \argmin_{\mathbf{w}, b}\
\sum_{(\phi, U) \in \mathcal{D}}
\ell\bigl(\sigma(\mathbf{w}^\top \phi + b),\, U\bigr)
+ \lambda \|\mathbf{w}\|_1
\label{eq:lasso}
\end{equation}
where $\ell$ is the binary cross-entropy loss and
$\lambda$ controls sparsity over $\phi_{\text{cand}}$ so that
uninformative features drop out. The deployed gate is
\begin{equation}
  g(s) = \mathbf{1}\!\left[\sigma(\mathbf{w}^{*\top} \phi(s) + b^*)
  > \tau\right],
  \label{eq:gate}
\end{equation}
with threshold $\tau$ set by cross-validation on $\mathcal{D}$.
At deployment, the gate requires only a feature evaluation and a
single sigmoid, adding no extra LLM calls.

\paragraph{Signed weights as a diagnostic.}
The $\ell_1$ penalty in Eq.~\ref{eq:lasso} simultaneously
performs feature selection and direction discovery. The sign of
each fitted weight provides an interpretable per-environment
estimate of the Two-Source mixture of \S\ref{sec:toy-model}:
$w_i > 0$ indicates the environment sits on the Type~D side of
the reversal threshold $p_I^*$ for signal~$i$ (high signal
$\to$ trigger), $w_i < 0$ the Type~I side (high signal $\to$
\emph{don't} trigger), and $w_i \approx 0$ an uninformative
signal. The complete pipeline, including online adaptation for
drifting environments and detailed cost accounting, is provided
in Appendix~\ref{app:dial-details}.
